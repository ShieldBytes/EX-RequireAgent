# 渐进式披露文档结构说明

## 设计原则

所有项目统一采用渐进式披露结构，不论项目规模大小。读者可以从顶层快速获取全貌，按需下钻到模块细节。

## 目录结构

```
{output_dir}/
├── requirement-overview.md          # L0 执行摘要 + L1 功能全景 + L3 附录
├── modules/
│   ├── 01-用户管理.md               # L2 模块详情
│   ├── 02-订单系统.md
│   ├── 03-支付模块.md
│   └── ...
├── changelog.md                     # 变更记录
├── report.md                        # 优化报告
└── open-questions.md                # 开放问题
```

## 披露层级

| 层级 | 位置 | 内容 | 目标读者 | 阅读时间 |
|------|------|------|---------|---------|
| L0 执行摘要 | requirement-overview.md §1 | 产品定位 + 核心指标 | 所有人 | 30 秒 |
| L1 功能全景 | requirement-overview.md §2 | 模块总览 + 依赖 + 旅程 + 非功能概览 | PM/决策者 | 3 分钟 |
| L2 模块详情 | modules/*.md | 用户故事 + 详细需求 + 验收标准 + 异常场景 | 开发/QA | 按需阅读 |
| L3 附录 | requirement-overview.md §3-5 | 术语表 + 追溯矩阵 + 变更记录 | 审计/回溯 | 按需查阅 |

## 模块文件命名

格式：`{序号}-{模块名}.md`，如 `01-用户管理.md`、`02-订单系统.md`。

序号按优先级或功能域排列，保持阅读顺序的一致性。

## 模块生命周期

每个模块独立维护状态，状态流转如下：

```
草稿 → 优化中 → 待审核 → 已审核 → 已锁定
```

### 状态说明

| 状态 | 含义 | 允许的操作 |
|------|------|-----------|
| 草稿 | 初始生成，尚未优化 | 可自由编辑 |
| 优化中 | 正在进化迭代中 | Agent 和用户均可修改 |
| 待审核 | 达到目标分数，等待用户确认 | 用户审核，可打回 |
| 已审核 | 用户确认通过 | 仅允许小幅修正 |
| 已锁定 | 最终锁定，不再修改 | 需解锁后才能编辑 |

### 状态流转规则

1. 新创建的模块初始状态为**草稿**
2. 进入进化流程后自动变为**优化中**
3. 所有维度评分达到目标线后进入**待审核**
4. 用户确认通过后变为**已审核**
5. 用户主动锁定后变为**已锁定**
6. 用户可将**已审核**或**已锁定**的模块打回到**优化中**
