# 模块化文档拆分说明

## 目录结构

当需求文档采用模块化模式时，目录结构如下：

```
project/
├── requirement-overview.md          # 主文档（概览）
└── modules/
    ├── module-01-用户管理.md         # 模块详细需求
    ├── module-02-订单系统.md
    ├── module-03-支付模块.md
    └── ...
```

- **主文档**：只包含产品概述、模块总览表、依赖关系、术语表和全局规则
- **模块文档**：包含该模块的详细功能需求、接口定义、业务规则和验收标准

## 拆分触发条件

| 功能点数 | 策略 | 说明 |
|---------|------|------|
| ≤ 3 | 单文件 | 需求较少，保持单文件即可 |
| 4 - 8 | 建议拆分 | 系统提示用户是否拆分为模块化结构 |
| > 8 | 自动拆分 | 自动采用模块化结构，按功能域拆分 |
| 文档超 50 页 | 触发建议 | 无论功能点数多少，文档过长时触发拆分建议 |

## 模块生命周期

每个模块独立维护状态，状态流转如下：

```
草稿 → 优化中 → 待审核 → 已审核 → 已锁定
```

### 状态说明

| 状态 | 含义 | 允许的操作 |
|------|------|-----------|
| 草稿 | 初始生成，尚未优化 | 可自由编辑 |
| 优化中 | 正在进化迭代中 | Agent 和用户均可修改 |
| 待审核 | 达到目标分数，等待用户确认 | 用户审核，可打回 |
| 已审核 | 用户确认通过 | 仅允许小幅修正 |
| 已锁定 | 最终锁定，不再修改 | 需解锁后才能编辑 |

### 状态流转规则

1. 新创建的模块初始状态为**草稿**
2. 进入进化流程后自动变为**优化中**
3. 所有维度评分达到目标线后进入**待审核**
4. 用户确认通过后变为**已审核**
5. 用户主动锁定后变为**已锁定**
6. 用户可将**已审核**或**已锁定**的模块打回到**优化中**
