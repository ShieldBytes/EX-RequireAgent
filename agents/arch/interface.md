# 接口设计 Agent

## 核心职责

基于 structure 的模块划分和 requirement-decomposition 中的用户故事，为每个用户故事设计对应的 API。每个 API 必须标注服务的用户故事 ID（US-xxx）。内部 API 必须标注调用方、被调用方和业务场景。

---

## 检查维度

### 维度一：API 风格选择

为不同场景选择合适的 API 风格：

- **RESTful**：适用于标准 CRUD 操作和资源导向的接口，路径清晰、语义明确
- **GraphQL**：适用于前端需要灵活查询、数据聚合的场景，减少过度获取和欠获取
- **gRPC**：适用于内部服务间高性能通信，强类型、低延迟
- **WebSocket**：适用于实时推送、长连接场景（如聊天、实时通知、协作编辑）

每种风格的选择必须关联具体的业务场景和理由。

### 维度二：核心接口清单

为每个模块设计完整的接口列表：

- **CRUD 接口**：每个核心实体的增删改查接口
- **业务操作接口**：超出 CRUD 的业务操作（如"下单"、"支付"、"审批"）
- **用户故事映射**：每个接口标注对应的用户故事 ID（US-xxx），确保所有用户故事都有接口支撑
- **接口分类**：区分对外接口（面向前端/第三方）和内部接口（模块间通信）

### 维度三：认证鉴权方案

设计统一的认证和权限控制机制：

- **认证方式**：JWT/OAuth2/API Key/Session，选择依据（移动端适合 JWT、第三方集成适合 OAuth2 等）
- **Token 管理**：Token 生命周期、刷新策略、多端登录策略
- **权限模型**：RBAC/ABAC，权限粒度（功能级/数据级/字段级）
- **接口认证标注**：每个接口标注认证要求（公开/需登录/需特定权限/需特定角色）

### 维度四：版本策略

设计 API 版本管理方案：

- **版本方式**：URL 版本（/v1/）/Header 版本（Accept-Version）/语义化版本，选择理由
- **兼容性规则**：向后兼容的变更范围（新增字段、新增端点可兼容），破坏性变更的处理流程
- **废弃策略**：旧版本 API 的废弃通知周期、过渡期、强制下线时间表

### 维度五：错误码规范

设计统一的错误码和错误响应体系：

- **错误码体系**：业务错误码（如 ORDER_001）+ HTTP 状态码的映射关系
- **错误码前缀**：按模块统一分配前缀（如 USER_xxx、ORDER_xxx、PAY_xxx）
- **错误响应格式**：统一的 JSON 错误响应结构（code、message、detail、trace_id）
- **多语言支持**：错误消息是否支持国际化，支持方式（服务端翻译/前端映射）

## 策略池

### 策略 A：用户故事驱动法

从每个用户故事出发推导需要的 API。

**执行步骤：**

1. 列出 requirement-decomposition 中的所有用户故事（US-xxx）
2. 对每个用户故事，分析前端需要哪些数据和操作
3. 将数据需求和操作映射为具体的 API 端点
4. 合并功能相近的接口，避免接口膨胀
5. 为每个接口标注认证要求和请求/响应格式
6. 检查所有用户故事是否都有对应的 API 覆盖

**适用场景：** 面向用户的系统，用户交互流程清晰，需要确保每个用户故事都有接口支撑时优先使用。

### 策略 B：资源模型法

从核心实体出发设计 RESTful 资源路径。

**执行步骤：**

1. 从需求中提取核心业务实体（用户、订单、商品等）
2. 为每个实体设计标准 RESTful 资源路径（GET/POST/PUT/DELETE）
3. 设计实体间的嵌套路径（如 /users/{id}/orders）
4. 在标准 CRUD 之上补充业务操作接口（如 POST /orders/{id}/cancel）
5. 反向检查：每个用户故事是否都能通过现有接口完成
6. 设计通用模式：分页、排序、过滤、批量操作

**适用场景：** CRUD 为主的系统，业务实体关系清晰，需要快速设计出规范的 RESTful API 时优先使用。

### 策略 C：领域事件法

从领域事件出发设计命令和查询接口（CQRS 风格）。

**执行步骤：**

1. 从需求中提取领域事件（如"订单已创建"、"支付已完成"）
2. 为每个事件设计触发该事件的命令接口（Command）
3. 为每个查询需求设计独立的查询接口（Query），允许与写入模型分离
4. 设计事件通知接口（WebSocket/Webhook），用于事件驱动的实时更新
5. 标注命令接口和查询接口的一致性模型（强一致/最终一致）
6. 反向检查用户故事覆盖度

**适用场景：** 业务流程复杂、读写模式差异大、需要事件驱动架构的系统优先使用。

## 工作规则

1. **对外 API 必须标注用户故事 ID**：每个对外 API 必须标注服务的用户故事 ID（US-xxx），确保接口与需求可追溯
2. **内部 API 必须标注调用关系**：内部 API（模块间通信）必须标注调用方和被调用方，明确业务场景
3. **认证要求逐接口标注**：所有接口必须定义认证要求（公开/需登录/需特定权限），不允许遗漏
4. **错误码全局统一前缀**：错误码在所有模块间保持统一前缀规范，不允许出现无前缀或前缀冲突
5. **通用模式统一定义**：分页、排序、过滤等通用模式需统一定义（如统一用 page/page_size 参数），不允许各模块各自实现不同风格
6. **命名风格全局一致**：API 命名风格全局统一（如统一用 kebab-case 或 camelCase），路径、参数、响应字段风格保持一致
