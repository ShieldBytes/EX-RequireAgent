# 优化引擎：能量管理与高级机制

本模块定义编排器的高级控制机制，包括能量管理、分数追踪、时间预算、复杂度控制、信任度、疲劳检测、反群体思维、协作记忆、挑战追踪和 P2 积累告警。

---

## 一、能量阶段管理

编排器根据剩余可用轮次比例（当前轮次 / 最大轮次）动态调整行为策略：

| 能量区间 | 剩余轮次比例 | 策略 | 具体行为 |
|---------|------------|------|---------|
| 全力阶段 | 100%-50%（前半程） | 全力探索 | 调度所有已启用 Agent，每轮完整执行 3.1-3.8（含红队全流程） |
| 聚焦阶段 | 50%-20% | 聚焦短板 | 只调度目标维度 Agent + red-team + writer，红队执行遗留复查+建议审查+系统性审查+追问 |
| 精简阶段 | 20%-10% | 精简流程 | 红队仍执行遗留复查+建议审查（不做系统性审查和关联推理），保留追问 |
| 终结阶段 | <10%（最后 1-2 轮） | 收尾终结 | 红队只做遗留复查（不做新挑战），如果 open_challenges 全部 resolved 则跳过红队 |

**计算公式**：
```
能量值 = (max_rounds - current_round) / max_rounds * 100
```

**行为切换**：
- 进入聚焦阶段时提示用户："进入聚焦模式，集中处理最薄弱维度"
- 进入精简阶段时提示用户："进入精简模式，优先保证核心质量"
- 进入终结阶段时提示用户："即将达到最大轮次，进行最后修正"

## 二、分数速度追踪

追踪每轮的分数变化速度，用于预测和决策：

```
分数速度 = (本轮目标维度分数 - 上轮目标维度分数) / 1轮

加速度 = 本轮速度 - 上轮速度
```

**应用规则**：
1. **速度为正且加速**：优化效果良好，继续当前策略
2. **速度为正但减速**：边际效益递减，考虑切换策略或维度
3. **速度为零**：停滞信号，连续 2 轮标记为 stalled
4. **速度为负**：退化信号，立即回滚

**预测**：
- 基于当前速度预测达标所需轮次：`预计剩余轮次 = (target_score - current_score) / 分数速度`
- 如果预计轮次 > 剩余轮次，向用户提示："按当前速度，{维度名} 可能无法在剩余轮次内达标"

---

## 三、时间预算管理

编排器需要追踪时间消耗，确保流程不会无限期运行：

**计时规则**：
- Agent 执行时间计入预算（从发送 SendMessage 到收到响应）
- 用户思考/回复的等待时间不计入预算
- 文件读写操作时间不计入预算

**预算分配**（如设定总预算）：
| 阶段 | 建议占比 |
|------|---------|
| 阶段零（初始化） | 5% |
| 阶段一（前置侦察） | 10% |
| 阶段二（广度扫描） | 25% |
| 阶段三（深度优化） | 50% |
| 阶段四+五（终审+交付） | 10% |

**注意**：当前版本不强制时间预算，仅供监控参考。编排器可在每轮结束时记录累计时间到 state.json。

---

## 四、渐进式复杂度定义

根据项目规模和用户需求，编排器支持三种复杂度模式：

| 模式 | 触发条件 | Agent 数量 | 功能范围 |
|------|---------|-----------|---------|
| 极简模式 | raw_input < 50 字 或 用户指定 `--simple` | 6 个核心 Agent | writer, evaluator, completeness, consistency, user-journey, feasibility |
| 标准模式 | 默认 | 全部已启用 Agent | 完整流程，含 red-team 和按需 Agent |
| 专家模式 | 用户指定 `--expert` | 全部 Agent + 高级功能 | 启用所有按需 Agent，启用自动策略生成，启用收敛信号放大 |

**模式选择**：
- 极简模式跳过：red-team 挑战、按需 Agent 建议、进化日志
- 标准模式为默认行为
- 专家模式额外启用：所有按需 Agent、require-engine 高级机制、require-agent-protocol 全部规则

## 五、Agent 信任度计算

编排器为每个 Agent 维护一个信任度分数，用于调整其发现的权重：

**初始信任度**：所有 Agent 初始信任度 = 1.0

**信任度更新规则**：
```
每轮结束后：
  如果该 Agent 的发现被 writer 采纳且评分提升：
    信任度 += 0.1（上限 1.5）
  如果该 Agent 的发现被采纳但评分未变或下降：
    信任度 -= 0.1（下限 0.5）
  如果该 Agent 的发现被 red-team 否决：
    信任度 -= 0.05（下限 0.5）
```

**信任度应用**：
- 信任度 >= 1.2 的 Agent：其 P2 级别发现升级为 P1 处理
- 信任度 <= 0.7 的 Agent：其 P1 级别发现降级为 P2 处理
- 信任度记录在 state.json 的 `agent_trust` 字段中

## 六、Agent 疲劳检测

当某 Agent 连续多轮产出重复内容时，判定为疲劳：

**检测规则**：
- 比较该 Agent 连续两轮的 findings
- 如果重复率 > 50%（关键词重叠判定），标记为疲劳

**疲劳处理**：
1. 刷新 Agent 上下文：在下次调度时追加指令：
   "注意：你前一轮的发现与上上轮高度重复。请换一个全新的角度重新审视文档，避免重复已提出的问题。"
2. 如果连续 3 轮疲劳：
   - 对于可选 Agent：暂停调度该 Agent 直到切换到新维度
   - 对于必需 Agent（如 writer、evaluator）：切换策略
3. 疲劳状态记录在 state.json 中

## 七、反群体思维机制

防止所有 Agent 趋同，丧失多样性视角：

**触发规则**：

固定频率：每 2 轮执行一次（从 3 轮缩短为 2 轮）

条件触发：连续 2 轮所有维度分数都上升（无一下降或持平）→ 额外触发一次
理由：越顺利越要警惕，所有分数同时上升可能是群体思维的表现

质疑范围扩大：不只质疑"已达标维度"，也质疑"连续 2 轮上升的维度"

**执行方式**：
1. 从当前文档中随机选取一个已达标维度（分数 >= target_score 的维度）或连续 2 轮上升的维度
2. 向该维度对应的 Agent 发送质疑指令：
   "你上一次对这个维度评价较高。请用批判性思维重新审视：有没有被你遗漏的问题？有没有因为其他维度的改进而引入的新问题？"
3. 如果 Agent 确实发现新问题 -> 记录为额外发现，供 writer 参考
4. 如果 Agent 没有新发现 -> 确认该维度稳固，不做操作

**注意**：反群体思维不影响正常的优化流程，只是额外的检查步骤。

## 八、Agent 协作组合记忆

记录不同 Agent 组合调度的效果，用于优化未来调度策略：

**记录格式**（追加到 state.json 的 `combo_memory`）：

```json
{
  "combo_memory": [
    {
      "round": "{轮次}",
      "agents_used": ["{agent1}", "{agent2}"],
      "target_dimension": "{目标维度}",
      "score_change": "{分数变化}",
      "effective": true
    }
  ]
}
```

**应用规则**：
- 当选择策略时，查看 combo_memory 中相同维度的历史记录
- 优先使用历史上 effective=true 的 Agent 组合
- 避免使用历史上 effective=false 的组合（除非已切换策略）
- 结合 `evolution/strategies/effectiveness.json` 的跨项目数据

**跨项目积累**：
- 项目完成时，将 combo_memory 精华写入进化日志
- 下次相似项目加载时，可参考历史 Agent 组合效果

## 九、挑战追踪机制

### open_challenges 结构

在 state.json 中维护挑战遗留清单：

```json
{
  "open_challenges": [
    {
      "id": "RT-001",
      "priority": "P0",
      "status": "unresolved",
      "target": "优惠券模块 — 领取规则",
      "challenge": "未定义'每人'的判定依据",
      "perspective": "恶意用户",
      "first_round": 2,
      "last_round": 3,
      "rounds_open": 2,
      "escalated": false,
      "writer_rejection_count": 0,
      "writer_rejection_reason": null
    }
  ]
}
```

### challenge_coverage 结构

在 state.json 中维护挑战覆盖矩阵：

```json
{
  "challenge_coverage": {
    "01-用户模块": {
      "恶意用户": { "status": "resolved", "round": 2 },
      "刁钻甲方": { "status": "unchallenged" },
      "需求质检员": { "status": "unchallenged" }
    },
    "02-订单模块": {
      "恶意用户": { "status": "unchallenged" },
      "刁钻甲方": { "status": "partial", "round": 3 },
      "需求质检员": { "status": "unchallenged" }
    }
  }
}
```

非模块化模式下，以功能点名称代替模块名作为 key。

### 更新规则

每轮结束后（步骤 3.8）：
1. open_challenges 中 resolved 的 → 移出
2. partial/unresolved 的 → rounds_open +1
3. 覆盖矩阵：红队审查了的 → 更新状态；本轮被修改的功能点 → 所有视角重置为 unchallenged

### 升级报告规则

- P0 连续 2 轮 unresolved 且 escalated=false → escalated=true，向用户报告
- P1 连续 3 轮 unresolved 且 escalated=false → escalated=true，向用户报告
- 报告格式：
  "以下问题已持续 {N} 轮未解决，可能需要人工决策：
   [{优先级}] {目标}：{挑战内容}
   最近回应：{上轮回应摘要}"

### Writer 否决分歧升级

当同一条挑战连续 2 轮被 writer 标记为 deferred_by_writer，但红队仍标记为 P0/P1：
- 向用户报告分歧，请求人工裁决
- 不自动偏向任何一方

### 覆盖率统计

每轮进度展示中追加覆盖率：
已覆盖 = resolved 数 / 总(功能点 × 视角数) × 100%

## 十、P2 积累告警

### 告警规则

每轮结束后统计 open_challenges 中 status != resolved 的 P2 数量：

- P2 未解决数 >= 5 → 向用户提示：
  "累积了 {N} 个中等优先级未解决问题，建议安排一轮专项清理。
   是否在下一轮集中处理 P2 问题？(Y/N)"
  用户选 Y → 下一轮为"P2 清理轮"（见 require-optimize.md 步骤 3.1）

- P2 未解决数 >= 10 → 强制提醒：
  "已积压 {N} 个中等问题，质量风险较高。建议优先清理。"

### P2 清理轮行为

P2 清理轮不走正常的"选最低分维度"逻辑，而是：
1. 将所有积压的 P2 建议合并
2. 直接调度 writer 批量整合
3. 调度 evaluator 重新评分
4. 不调度维度 Agent 和红队（已有 P2 建议，不需要再分析）
