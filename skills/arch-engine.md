# 架构优化引擎：能量管理与高级机制

本模块定义架构编排器的高级控制机制，包括能量管理、分数追踪、Agent 信任度、疲劳检测、反群体思维、收敛信号放大、Agent 组合记忆、维度映射、渐进式复杂度和智能经验加载。

---

## 一、能量阶段管理

编排器根据剩余可用轮次比例（当前轮次 / 最大轮次）动态调整行为策略：

| 能量区间 | 剩余轮次比例 | 策略 | 具体行为 |
|---------|------------|------|---------|
| 全力阶段 | 100%-50%（前半程） | 全面挑战 | 四视角全量挑战 + 级联检查 + 覆盖验证，完整执行 3.1-3.12 |
| 聚焦阶段 | 50%-20% | 聚焦短板 | 只挑战目标维度 + 主责 Agent + writer，跳过级联检查和覆盖验证 |
| 精简阶段 | 20%-10% | 精简修正 | 跳过追问（3.4），只做主动改进（3.2）+ writer（3.8）+ evaluator（3.9） |
| 终结阶段 | <10%（最后 1-2 轮） | 收尾 | 只做最小修正，准备进入终审 |

**计算公式**：
```
能量值 = (max_rounds - current_round) / max_rounds * 100
```

**行为切换**：
- 进入聚焦阶段时提示用户："进入聚焦模式，集中处理最薄弱维度"
- 进入精简阶段时提示用户："进入精简模式，跳过追问环节，直接改进"
- 进入终结阶段时提示用户："即将达到最大轮次，进行最后修正"

## 二、分数速度追踪

追踪每轮的分数变化速度，用于预测和决策：

```
分数速度 = (本轮目标维度分数 - 上轮目标维度分数) / 1轮

加速度 = 本轮速度 - 上轮速度
```

**应用规则**：
1. **速度为正且加速**：优化效果良好，继续当前策略
2. **速度为正但减速**：边际效益递减，考虑切换策略或维度
3. **速度为零**：停滞信号，连续 2 轮标记为 stalled
4. **速度为负**：退化信号，立即回滚

**预测**：
- 基于当前速度预测达标所需轮次：`预计剩余轮次 = (target_score - current_score) / 分数速度`
- 如果预计轮次 > 剩余轮次，向用户提示："按当前速度，{维度名} 可能无法在剩余轮次内达标"

## 三、Agent 信任度

编排器为每个 Agent 维护一个信任度分数，用于调整其发现的权重。

**初始信任度**：所有 Agent 初始信任度 = 1.0

**信任度更新规则**：
```
每轮结束后：
  如果该 Agent 的发现被采纳且目标维度评分提升：
    信任度 += 0.1（上限 1.5）
  如果该 Agent 的发现被采纳但评分未变或下降：
    信任度 -= 0.1（下限 0.5）
  如果该 Agent 的发现被 challenger 推翻：
    信任度 -= 0.1（下限 0.5）
```

**信任度应用**：
- 信任度 >= 1.2 的 Agent：其建议优先采纳，challenger 对其挑战需更充分理由
- 信任度 <= 0.7 的 Agent：其建议需经过额外验证才能采纳
- 信任度记录在 state.json 的 `agent_trust` 字段中

## 四、疲劳检测

检测 challenger 重复挑战和生成 Agent 重复修改的情况。

**检测规则**：
- 比较 challenger 连续两轮的挑战列表，如果关键词重叠率 > 50%，标记 challenger 疲劳
- 比较生成 Agent 连续两轮的修改方案，如果关键词重叠率 > 50%，标记该 Agent 疲劳

**疲劳处理**：
1. 刷新 Agent 上下文：在下次调度时追加指令：
   "注意：你前一轮的输出与上上轮高度重复。请换一个全新的角度重新审视，避免重复已提出的内容。"
2. 如果连续 3 轮疲劳：
   - 对于 challenger：跳过全量挑战，只做主动改进
   - 对于生成 Agent：切换策略（从映射表中选择备选策略）
3. 疲劳状态记录在 state.json 的 `fatigue_status` 字段中

## 五、反群体思维

防止所有 Agent 趋同，丧失多样性视角。

**触发规则**：每 2 轮执行一次反向质疑

**执行方式**：
1. 从当前架构中选取一个已达标维度（分数 >= target_score 的维度）的设计决策
2. 向该维度对应的主责 Agent 发送质疑指令：
   "你上一次对这个维度的设计评价较高。请用批判性思维重新审视：有没有被你遗漏的问题？有没有因为其他维度的改进而引入的新问题？当前设计是否存在隐藏的技术债？"
3. 如果 Agent 确实发现新问题 -> 记录为额外发现，供下一轮挑战参考
4. 如果 Agent 没有新发现 -> 确认该维度稳固，不做操作

**注意**：反群体思维不影响正常的挑战流程，只是额外的检查步骤。

## 六、收敛信号放大

当 challenger 的多个挑战指向同一设计缺陷时，放大该信号。

**检测规则**：
- 分析 `challenge_list` 中的所有挑战项
- 如果 2 个及以上挑战项的根因指向同一设计决策或同一组件 -> 标记为关键问题

**放大处理**：
1. 将该设计决策/组件标记为 `critical_issue`
2. 无论原始严重程度如何，升级为 P0 处理
3. 在调度主责 Agent 回应时，额外强调："这是多个挑战项共同指向的关键问题，请优先处理"
4. 关键问题记录在 state.json 的 `critical_issues` 字段中

## 七、Agent 组合记忆

记录不同 Agent 级联组合的效果，用于优化未来调度策略。

**记录格式**（追加到 state.json 的 `combo_memory`）：

```json
{
  "combo_memory": [
    {
      "round": "{轮次}",
      "primary_agent": "{主责 Agent}",
      "secondary_agent": "{辅助 Agent}",
      "cascade_agents": ["{级联 Agent 列表}"],
      "target_dimension": "{目标维度}",
      "score_change": "{分数变化}",
      "effective": true
    }
  ]
}
```

**应用规则**：
- 当选择策略时，查看 combo_memory 中相同维度的历史记录
- 优先使用历史上 effective=true 的 Agent 组合
- 避免使用历史上 effective=false 的组合（除非已切换策略）
- 结合 `evolution/arch-strategies/effectiveness.json` 的跨项目数据

**跨项目积累**：
- 项目完成时，将 combo_memory 精华写入进化日志
- 下次相似项目加载时，可参考历史 Agent 组合效果

## 八、维度→Agent→策略 三角映射

完整的维度、主责 Agent、辅助 Agent 与可用策略映射表：

| 维度 | 主责 Agent | 辅助 Agent | 可用策略 |
|------|----------|----------|---------|
| 可扩展性 | structure | storage | structure-A（分层架构）, structure-B（微服务拆分）, storage-A（分库分表）, storage-B（缓存策略）, storage-C（读写分离） |
| 灵活性 | structure | interface | structure-A（分层架构）, structure-B（微服务拆分）, interface-A（API 网关）, interface-B（协议适配） |
| 可维护性 | structure | platform | structure-A（分层架构）, structure-B（微服务拆分）, platform-A（CI/CD 流水线） |
| 可靠性 | structure | storage | structure-C（容错设计）, storage-C（读写分离/备份） |
| 安全性 | interface | platform | interface-A（API 网关/鉴权）, platform-A（安全基线） |
| 成本效率 | platform | structure | platform-C（资源优化）, structure-C（容错设计/降级） |
| 可观测性 | platform | — | platform-A（监控告警）, platform-B（日志追踪）, platform-C（资源监控） |
| 覆盖度 | coverage | — | — |

**策略选择优先级**：
1. 查看 combo_memory 中同维度的历史有效策略
2. 查看 evolution/arch-strategies/ 中的跨项目有效策略
3. 使用默认策略（映射表中排第一的策略）

## 九、渐进式复杂度

根据项目规模和用户需求，编排器支持三种复杂度模式：

| 模式 | 触发条件 | Agent 数量 | 功能范围 |
|------|---------|-----------|---------|
| 极简模式 | 需求 < 50 字 或 用户指定 `--simple` | 5 个核心 Agent | structure + platform + interface + writer + evaluator |
| 标准模式 | 默认 | 全部已启用 Agent | 完整流程，含 challenger 和所有引擎机制 |
| 专家模式 | 用户指定 `--expert` | 全部 Agent + 高级功能 | 全 Agent + 全引擎机制 + 收敛信号放大 + 反群体思维 |

**模式选择**：
- 极简模式跳过：challenger 挑战、级联检查、覆盖验证、进化日志
- 标准模式为默认行为
- 专家模式额外启用：所有可选 Agent、全部引擎高级机制、架构模式匹配

## 十、智能经验加载

在项目启动时，从进化数据中加载相关历史经验。

**加载来源**：`evolution/arch-strategies/` 目录

**过滤规则**：
1. 按项目类型过滤（project_type 匹配）
2. 按团队规模过滤（team_size 在 0.5x-2x 范围内）
3. 按技术栈过滤（tech_stack 有交集）

**多因子推荐排序**：
```
推荐分 = 项目类型匹配度 * 0.4
       + 技术栈重叠度 * 0.3
       + 用户满意度 * 0.2
       + 时效性（越近越高） * 0.1
```

**争议策略检测**：
- 如果某策略在历史中既有 effective=true 也有 effective=false 记录
- 标记为争议策略，展示给用户让其决定是否采用
- 展示格式：`策略 {名称}：成功 {N} 次，失败 {M} 次，建议 {采用/慎用}`

**架构模式匹配**：
- 从 `evolution/arch-strategies/patterns.json` 中查找同类项目的历史架构
- 如果找到匹配度 > 0.7 的历史项目，直接展示为参考：

```
找到相似项目架构参考：

项目：{历史项目名}
类型：{project_type}
规模：{scale}
架构模式：
  - 部署方式：{deployment}
  - 模块划分：{modules}
  - 技术栈：{tech_stack}
  - 关键决策：{key_decisions}
  - 月成本：{monthly_cost}
  - 用户满意度：{user_satisfaction}/10

是否参考此架构作为起点？
```

- 如果用户同意，将历史架构作为初始方案的参考输入
- 如果用户拒绝，从零开始设计
