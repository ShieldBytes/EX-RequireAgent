# 阶段四：终审 + 阶段五：交付输出

本模块包含终审（阶段四）和交付输出（阶段五）的核心步骤。

---

## 阶段四：终审

### 步骤 4.1：最终评分 — 调度 arch-evaluator

使用 `SendMessage` 向 **arch-evaluator** Agent 发送以下指令：

> **输入**：将 overview 文件、所有 module 文件和 decisions/ 下所有 ADR 的完整内容附在消息中。
>
> **指令**：
> "你是架构评估 Agent。这是最终一轮评审，请对以下架构文档做全面终审评分。
>
> 按照以下 8 个维度评分：可扩展性、灵活性、可维护性、可靠性、安全性、成本效率、可观测性、需求覆盖度
>
> 评分层级：模块级 + 项目级
>
> 除了维度评分外，请额外检查以下终审项目：
> 1. **追溯矩阵完整性**：每条需求是否都有对应的架构组件，每个架构组件是否都有需求支撑
> 2. **ADR 与文档一致性**：decisions/ 中的决策记录是否与架构文档描述一致，有无矛盾
> 3. **Mermaid 图与文字描述一致性**：架构图中的组件、连接是否与文字描述完全吻合
> 4. **术语统一**：overview 和各 module 文件是否使用了统一的术语，无混用现象
> 5. **模块总览表与 modules/ 文件一致**：overview 中的模块总览表是否与各 module 文件的实际内容一致
>
> 如果发现终审问题，在优化建议部分指出，并标明问题出现在 overview、哪个 module 文件还是哪个 ADR。
>
> 架构概览：
> {overview 内容}
>
> 模块文件：
> {逐个列出每个 module 文件名和内容}
>
> ADR：
> {decisions/ 目录下所有文件内容}"

**输出处理**：
1. 解析 8 维度 × 两层的最终评分
2. 保存到 `.arch-agent/projects/{项目名}/scores/final.json`

### 步骤 4.2：如果终审发现问题 — 最终修正

如果 evaluator 在终审中指出了追溯矩阵、ADR 一致性、Mermaid 图、术语、模块总览表方面的具体问题：

使用 `SendMessage` 向 **arch-writer** Agent 发送修正指令：

> "你是架构写手 Agent。这是终审修正，请只修复以下具体问题，不要添加新内容、不要改变文档结构：
>
> {evaluator 指出的终审问题列表，包含各问题所在的文件}
>
> 在 overview 变更记录中记录：版本 v{final}，轮次 终审，来源 evaluator, writer。
>
> 当前 overview：
> {overview 内容}
>
> 当前模块文件：
> {逐个列出每个 module 文件名和内容}
>
> 当前 ADR：
> {decisions/ 目录内容}"

将修正后的文件保存为最终版本。

### 步骤 4.3：追溯矩阵终检 — 调度 arch-coverage

使用 `SendMessage` 向 **arch-coverage** Agent 发送以下指令：

> "请对架构文档进行最终双向验证：
>
> 1. **正向验证**：每条需求 → 是否有对应的架构模块/组件覆盖
> 2. **反向验证**：每个架构模块/组件 → 是否有需求支撑
>
> 需求文档：
> {需求文档内容}
>
> 架构文档：
> {完整架构文档}"

**输出处理**：如发现遗漏，记录到交付报告中（不再修改架构文档）。

## 阶段五：交付输出

| 指标 | 数值 |
|------|------|
| 总挑战轮次 | {轮次数} |
| P0 挑战总数 | {数量} |
| P1 挑战总数 | {数量} |
| P2 挑战总数 | {数量} |
| P3 挑战总数 | {数量} |

### 步骤 5.4：展示最终交付

```
架构文档优化完成！

输出文件：
  {output_dir}/
  ├── overview.md                  <- 架构概览
  ├── modules/
  │   ├── {模块名}.md              <- 模块详细设计
  │   └── ...
  ├── decisions/
  │   ├── ADR-001-{决策名}.md      <- 架构决策记录
  │   └── ...
  ├── challenge-report.md          <- 挑战报告
  ├── requirement-feedback.md      <- 需求反馈
  └── report.md                    <- 优化报告

阅读指南：
  overview.md                      -> 30 秒了解架构全貌
  modules/*.md                     -> 按需查看模块详细设计
  decisions/*.md                   -> 了解关键决策的权衡
  requirement-feedback.md          -> 需要需求侧关注的问题

工作区：
  .arch-agent/projects/{项目名}/
  ├── state.json                 <- 项目状态
  ├── versions/                  <- 所有版本快照
  └── scores/                    <- 所有轮次评分

后续可用操作：
  /arch --resume {项目名}        <- 恢复优化
  /arch --file <路径>             <- 基于文件重新开始
```

### 步骤 5.5：收集用户满意度

向用户提问：

```
请对本次架构设计结果打分（1-10 分），并提供反馈（可选）：
```

等待用户回复，解析评分和反馈内容。

### 步骤 5.7：自动推送

将进化数据推送到共享仓库（如已配置）：

1. 检查是否配置了进化数据共享仓库
2. 如已配置，执行 `git add` + `git commit` + `git push`
3. 推送失败不影响主流程，仅记录警告

### 步骤 5.8：释放锁

删除 `.arch-agent/projects/{项目名}/lock.json`（如果存在）。

更新 `state.json`：`phase -> completed`。
