# 团队协作与锁机制

本模块定义项目锁机制和团队协作模式（--collab、--merge-input、--review、--role）。

---

## 一、锁机制

### 锁获取（步骤 0.2 之后执行）

检查 `.require-agent/projects/{项目名}/lock.json`：

**如果不存在** -> 创建锁：
```json
{
  "locked": true,
  "user": "current",
  "started": "{ISO 时间}",
  "last_heartbeat": "{ISO 时间}",
  "project": "{项目名}"
}
```

**如果存在**：
  读取 last_heartbeat，如果 > 2 分钟前 -> 视为过期，覆盖创建新锁
  否则 -> 提示用户：
  "另一个会话正在优化「{project}」。
   使用 --force 可强制接管。"
  如果用户未使用 --force -> 终止

### 锁释放（步骤 5.6）

在阶段五结束后删除 lock.json。
在错误处理中也尝试释放锁。

### 过期检测

- 每次编排器执行关键步骤时更新 `last_heartbeat`
- 其他会话检查锁时，如果 last_heartbeat 超过 2 分钟前，视为过期锁
- 过期锁可被任何新会话覆盖

## 二、--collab 模式

如果用户使用了 --collab：

1. 在 {output_dir}/team-input/ 下生成三个输入模板（基于 templates/team-input-template.md）：
   - pm-input.md（role: 产品经理）
   - dev-input.md（role: 开发）
   - design-input.md（role: 设计师）

2. 每个模板文件包含以下结构：
   ```markdown
   # 团队输入 — {角色名}

   ## 补充需求
   <!-- 请在此添加你认为缺失的需求 -->

   ## 质疑点
   <!-- 请在此提出你认为不合理或有疑问的地方 -->

   ## 约束条件
   <!-- 请在此添加技术/业务/时间等约束 -->

   ## 优先级调整
   <!-- 如果你认为某些功能优先级需要调整，请在此说明 -->
   ```

3. 提示："团队输入模板已生成到 {output_dir}/team-input/，请各角色填写后运行 /require --merge-input"

4. 终止流程等待填写

---

## 三、--merge-input 模式

如果用户使用了 --merge-input：

1. 读取 {output_dir}/team-input/ 下所有文件
2. 解析每个文件中的补充需求、质疑点、约束
3. 合并逻辑：
   - **共识**（多人提到的相同/相似需求）-> 直接纳入 raw_input
   - **分歧**（有人赞成有人反对的点）-> 标记 [团队分歧待讨论]
   - **约束**（所有人提出的约束条件）-> 合并去重后纳入 raw_input
   - **优先级**（多人调整的优先级）-> 取众数或标记待讨论
4. 将合并结果作为 raw_input 继续正常流程（从步骤 0.2 开始）
5. 向用户展示合并摘要：
   ```
   团队输入合并完成：
   - 共识需求：{N} 条
   - 分歧点：{M} 条（已标记 [团队分歧待讨论]）
   - 约束条件：{K} 条
   ```

## 四、--review 模式

如果用户使用了 --review：

1. 读取 requirement-overview.md（及 modules/ 如模块化）
2. 扫描文档中的 `<!-- @review xxx: yyy -->` 标注
   - xxx 为评审人标识
   - yyy 为评审意见
3. 提取评审意见列表，分类：
   - 改进建议（含"建议"、"应该"等关键词）
   - 问题指出（含"问题"、"错误"、"矛盾"等关键词）
   - 确认通过（含"OK"、"同意"、"没问题"等关键词）
4. 将改进建议和问题指出作为优化输入，调度 Agent 处理：
   - 改进建议 -> 作为额外的 optimization_findings 传给 writer
   - 问题指出 -> 作为额外的 redteam_challenges 传给 writer
5. 冲突意见（不同评审人对同一处有矛盾意见）标记为决策点 [评审冲突待决策]
6. 向用户展示评审处理摘要

## 五、--role 接力模式

如果 --resume 带有 --role 参数：

根据角色调整优化偏好和达标线：

| 角色 | 侧重维度 | 达标线调整 |
|------|---------|-----------|
| pm（产品经理） | 业务闭环、用户旅程 | 这两个维度达标线 +1 |
| developer（开发） | 可行性、数据 | 这两个维度达标线 +1 |
| designer（设计师） | 用户旅程、无障碍与国际化 | 这两个维度达标线 +1 |

**执行方式**：
1. 读取当前 state.json 的 target_score
2. 对角色侧重的维度，将 target_score 临时 +1
3. 在步骤 3.1 选择目标维度时，使用调整后的达标线
4. 向用户提示："以 {角色} 视角继续优化，{维度1} 和 {维度2} 达标线提升至 {target_score+1} 分"
5. 其他维度保持原达标线不变

**接力流程**：
- 角色 A 完成后交付 -> 角色 B 用 `--resume {项目名} --role {角色}` 接力
- 每个角色的优化结果累积在同一个项目中
- round_history 中记录每轮的 `operator_role` 字段
