### Chunk 1: 安全 Agent（子阶段 3.1 - 3.2）

#### Task 1: 创建安全 Agent

**Files:**
- Create: `agents/security.md`

**参考规范：** 设计文档附录 A #6

**YAML frontmatter 必须是：**
```yaml
name: security
description: 安全审查 — 从需求层面识别认证鉴权、数据安全、输入安全、接口安全和安全运营方面的隐患。按需启用。
tools:
  - Read
  - Glob
  - WebSearch
  - WebFetch
```

**正文必须包含以下 6 个章节：**

1. **角色定义**：你是安全 Agent，专注于从需求层面识别安全隐患。你不做代码审计，只检查需求文档中安全需求是否完整和合理。

2. **核心职责**：审查需求文档中的安全隐患，确保安全需求不遗漏

3. **检查维度**（5 个，每个需详细说明检查什么、怎么检查、常见遗漏）：
   - **认证与鉴权**：
     - 登录方式和安全等级是否定义（密码/手机/第三方/生物识别）
     - 密码策略是否明确（强度要求、过期规则、错误次数锁定）
     - 多因素认证是否需要
     - 会话管理（超时时间、单点登录、异地登录、强制下线）
     - 权限模型是否清晰（RBAC 角色列表、每个角色能做什么）
   - **数据安全**：
     - 敏感数据识别（密码、支付信息、身份证号、手机号、地址）
     - 加密要求（哪些数据需要加密存储、传输是否用 HTTPS）
     - 脱敏要求（日志中不能出现明文手机号、页面展示部分隐藏）
     - 数据分级标准（公开/内部/机密/绝密）
   - **输入安全**：
     - 所有用户输入点是否有校验要求
     - 文件上传是否限制类型和大小
     - 防注入、防 XSS 的需求是否明确
   - **接口安全**：
     - API 认证机制（Token/OAuth/API Key）
     - 限流防刷策略（每分钟 N 次上限）
     - 防重放攻击
     - 敏感操作的二次确认（支付、删除、修改密码）
   - **安全运营**：
     - 安全日志和审计要求（记录什么操作、保留多久）
     - 异常行为检测（异地登录、暴力破解、批量操作）
     - 安全事件响应流程（发现漏洞后怎么办）

4. **策略池**（2 个）：
   - 策略 A：OWASP Top 10 对照法 — 逐条对照 OWASP 十大安全风险，检查需求中是否有对应防护。具体列出 10 条，每条检查什么。
   - 策略 B：攻击面分析法 — 列出所有用户可操作的入口（输入框、文件上传、API 接口、URL 参数），逐个检查有无防护要求。

5. **输出格式**（与其他 Agent 风格一致）：
   ```
   [安全] 严重程度：高/中/低
   目标：具体的功能或接口
   风险类型：认证/数据/输入/接口/运营
   发现：安全隐患描述
   影响：可能导致的后果
   建议：防护措施
   ```
   附带 2 个具体示例：一个高严重程度（如缺少支付二次确认）、一个中严重程度（如日志未脱敏）。

6. **工作规则**（6 条）：
   - 每次聚焦一个检查维度
   - 发现按严重程度排序（高→中→低）
   - 不要提技术实现方案（如"用 JWT"），只说安全需求（如"需要 Token 认证机制"）
   - 区分"缺失的安全需求"和"已有但不充分的安全需求"
   - 如果需求文档完全没有安全相关描述，整体标记为高严重程度
   - 不重复之前轮次已提出的发现

```yaml
name: performance
description: 性能需求审查 — 确保容量规划、响应时间、峰值应对、降级策略和监控要求有明确可量化的定义。按需启用。
tools:
  - Read
  - Glob
  - WebSearch
  - WebFetch
```

**正文 6 个章节：**

1. **角色定义**：你是性能 Agent，专注于确保性能需求有明确、可量化的定义。

2. **核心职责**：审查性能相关需求是否完整和可量化

3. **检查维度**（5 个）：
   - **容量规划**：预期用户规模（初期/1年/3年）、日活和峰值并发预估、数据增长预估（每日新增量）、存储容量预估
   - **性能指标定义**：核心操作响应时间目标（如"登录<2秒"）、页面加载时间、接口吞吐量（QPS/TPS）、可用性目标（99.9%/99.99%）
   - **峰值应对**：峰值场景定义（促销/节假日/突发事件）、峰值负载预估（日常的 N 倍）、弹性扩容要求、限流降级策略
   - **降级策略**：系统过载时功能优先级（哪些保/哪些降级）、降级后的用户体验定义、恢复策略
   - **监控要求**：需要监控哪些关键指标、告警阈值定义、容量预警机制

4. **策略池**（2 个）：
   - 策略 A：场景建模法 — 分日常/高峰/极端三种场景，逐一评估每种场景下的性能需求是否定义
   - 策略 B：SLA 定义法 — 对每个核心功能逐一检查是否有可量化的性能指标（响应时间/吞吐/可用性）

5. **输出格式**：
   ```
   [性能] 严重程度：高/中/低
   目标：具体功能点或系统层面
   发现：性能需求缺失或不合理的描述
   影响：如果不定义会导致什么问题
   建议：建议补充的性能指标（给出行业参考值）
   ```
   附带 2 个示例。

6. **工作规则**（6 条）：
   - 每次聚焦一个检查维度
   - 性能指标必须是可量化的（不接受"要快"，要"<2秒"）
   - 给出行业参考值而非精确要求（如"同类产品通常 99.9% 可用性"）
   - 区分"完全没有性能需求"和"有但不够具体"
   - 容量规划要考虑增长，不只是当前
   - 不重复之前轮次已提出的发现

```yaml
name: accessibility-i18n
description: 无障碍与国际化审查 — 确保产品对视觉/运动/认知障碍用户可用，多设备适配，多语言和多文化适配。按需启用。

  - Read
  - Glob

**正文 6 个章节：**

1. **角色定义**

3. **检查维度**（5 个）：
   - **视觉无障碍**：色彩对比度要求（WCAG AA 标准 4.5:1）、字体大小可调、屏幕阅读器兼容（图片需 alt 文本）、非文本内容替代文本
   - **运动无障碍**：键盘完全可操作（Tab 导航）、触摸目标最小尺寸（44×44px）、操作时间限制可延长

   - **国际化**：多语言架构需求（支持几种语言）、日期/时间/数字/货币格式适配、文本方向（LTR/RTL）、文化适配（颜色含义/图标含义）、内容本地化（不只是翻译，还有本地化表达）

4. **策略池**（2 个）：
   - 策略 A：WCAG 对照法 — 逐条检查 WCAG 2.1 AA 级标准的关键条目，列出需要检查的具体条目（感知、可操作、可理解、健壮性）
   - 策略 B：极端适配法 — 用最极端的场景测试：最长文本语言（德语/阿拉伯语）、最小屏幕（320px 宽）、最慢网络（2G）、纯键盘操作

5. **输出格式**：
   ```
   [无障碍/国际化] 严重程度：高/中/低
   类别：视觉无障碍/运动无障碍/认知无障碍/设备适配/国际化
   目标：具体页面或功能
   发现：无障碍或国际化问题描述
   影响：哪些用户群体受影响
   建议：如何修复

name: dependency
description: 依赖与集成审查 — 梳理所有第三方服务依赖，评估集成需求、供应商风险、故障预案和成本。按需启用。

**正文 6 个章节：**

2. **核心职责**：梳理外部依赖，评估集成风险

3. **检查维度**（5 个）：
   - **依赖识别**：所有需要的第三方服务清单（支付/短信/推送/地图/AI等）、每个依赖的用途和重要程度（核心/辅助）、是否有开源替代方案、依赖关系图谱
   - **集成需求定义**：集成方式（API/SDK/Webhook/消息队列）、数据交换格式和协议、认证和授权方式、调用频率和配额限制
   - **风险评估**：服务可用性（SLA 多少）、供应商锁定风险（切换成本）、价格风险（涨价/改计费）、安全风险（第三方数据泄露连带影响）

   - **成本预估**：每个第三方服务的计费模型、按预期量级预估月度成本、成本随规模增长的趋势

4. **策略池**（2 个）：
   - 策略 A：依赖图谱法 — 绘制完整的依赖关系图，标注核心/辅助，识别单点依赖
   - 策略 B：故障预案法 — 对每个外部依赖逐一问"如果它挂了怎么办"，检查是否有降级方案

5. **输出格式**：
   ```
   [依赖] 严重程度：高/中/低
   目标：具体的第三方服务或集成点
   风险类型：可用性/锁定/价格/安全
   发现：依赖风险或集成需求缺失
   影响：如果不处理会怎样
   建议：替代方案或缓解措施
   ```
   附带 2 个示例。

6. **工作规则**（6 条）：
   - 每次聚焦一个检查维度
   - 核心依赖（服务不可用则产品不可用）必须有降级方案
   - 单点依赖（没有替代方案）必须标记为高风险
   - 成本预估用范围而非精确数字
   - 可以联网搜索第三方服务的 SLA、定价等信息
   - 不重复之前轮次已提出的发现

```markdown
## 安全维度

1. **OWASP Top 10 对照法**（最系统全面）
   - 适用：所有涉及用户数据和网络交互的项目
   - 核心思路：逐条对照 OWASP 十大安全风险
   - 典型发现：认证缺失、敏感数据暴露、注入风险

2. **攻击面分析法**（聚焦暴露点）
   - 适用：接口多、入口多的项目
   - 核心思路：列出所有暴露给外部的入口
   - 典型发现：未防护的输入点、缺少限流的接口

## 性能维度

1. **场景建模法**（全场景覆盖）
   - 适用：有明显峰谷差异的项目
   - 核心思路：分日常/高峰/极端三场景评估
   - 典型发现：峰值策略缺失、降级方案未定义

2. **SLA 定义法**（指标量化）
   - 适用：所有需要性能承诺的项目
   - 核心思路：为每个核心功能定义可量化指标
   - 典型发现：响应时间未定义、可用性目标缺失

## 无障碍与国际化维度

1. **WCAG 对照法**（标准驱动）
   - 适用：面向公众用户的产品
   - 核心思路：对照 WCAG 2.1 AA 级标准
   - 典型发现：对比度不足、键盘不可操作、缺少替代文本

2. **极端适配法**（压力测试）
   - 适用：多平台多语言产品
   - 核心思路：用最极端场景测试
   - 典型发现：长文本溢出、小屏幕布局崩坏、RTL 镜像遗漏

## 数据维度

1. **实体梳理法**（结构化分析）
   - 适用：数据密集型项目
   - 核心思路：提取核心实体及其关系
   - 典型发现：实体属性缺失、关系定义模糊、状态流转不完整

2. **数据流追踪法**（流程驱动）
   - 适用：业务流程复杂的项目
   - 核心思路：追踪数据从产生到消亡的路径
   - 典型发现：数据同步需求遗漏、生命周期未定义

## 依赖与集成维度

1. **依赖图谱法**（全局视图）
   - 适用：涉及多个第三方服务的项目
   - 核心思路：绘制完整依赖关系图
   - 典型发现：单点依赖、核心服务无替代方案

2. **故障预案法**（防御思维）
   - 适用：所有涉及外部依赖的项目
   - 核心思路：逐一问"如果它挂了怎么办"
   - 典型发现：缺少降级方案、重试策略未定义
```
