# EX-RequireAgent 设计文档

> 需求自优化智能体 — 从模糊想法到精细 PRD 的自动迭代打磨

## 一、核心哲学

灵感来源于 [Karpathy/AutoResearch](https://github.com/karpathy/autoresearch) 的自主进化循环机制。

| AutoResearch | EX-RequireAgent |
|---|---|
| `program.md` — 人类定义研究方向 | `strategy.md` — 人类定义优化目标和偏好 |
| `train.py` — 唯一被迭代修改的产物 | 需求文档 — 唯一被迭代优化的产物 |
| 修改代码 → 训练 → 评估 → 保留/丢弃 | 优化需求 → 多维评审 → 评分 → 保留/回滚 |
| `val_bpb` 标准化指标 | 多维度评分体系（0-10） |
| 每轮固定 5 分钟时间预算 | 双重预算（轮次 + 时间） |
| 训练日志记录每轮变化 | 进化日志记录每轮优化策略和得分变化 |

**五大核心原则：**

1. **单一产物** — 所有 Agent 围绕同一份文档迭代，不产生散乱碎片
2. **可量化评估** — 每轮优化有明确分数，能比较好坏
3. **只进不退** — 只保留得分更高的版本，低于当前版本则丢弃
4. **自主循环** — 不需要人干预就能持续优化，直到达到阈值
5. **进化记忆** — 记录什么策略有效，跨项目积累经验

## 二、自适应进化引擎

### 2.1 三重约束的动态协作

评分、轮次、时间三者不是平级的 OR 关系，而是动态协作：

- **评分（Score）** — 大脑，决定优化**什么**
- **轮次（Rounds）** — 节奏，控制优化**多少次**
- **时间（Time）** — 安全网，确保**不失控**

```
时间预算（硬顶，不可突破，默认自动估算，可手动覆盖）
│
├── 阶段一：广度扫描（前 2-4 轮）
│   └── 每个维度都过一遍，建立基线分数
│
├── 阶段二：深度优化（第 N 轮起）
│   ├── 评分驱动：优先攻最低分维度
│   ├── 单维度连续 2 轮无提升 → 换策略，不是直接跳过
│   ├── 所有策略试完 → 尝试生成新策略
│   ├── 仍无效 → 标记该维度为"当前上限"，跳过
│   └── 剩余时间不足 → 压缩轮次，只攻最弱维度
│
└── 终止条件（任一触发）
    ├── 所有维度 ≥ 达标线 → ✅ 成功完成
    ├── 所有维度都已停滞 → ⏹ 提前退出
    ├── 轮次耗尽 → ⏹ 正常结束
    ├── 时间耗尽 → ⏹ 当前轮跑完后停止
    ├── ROI 过低 → ⏹ 主动建议停止
    └── 全局震荡无法收敛 → ⏹ 升级人工
```

### 2.2 分数速度驱动策略

不只看分数，还看分数变化速度：

| 速度 | 含义 | 对轮次的影响 | 对时间的影响 |
|---|---|---|---|
| 快速上升 | 优化策略很有效 | 继续当前维度，追加轮次 | 值得多投入时间 |
| 缓慢上升 | 有效但效率低 | 再给 1 轮机会 | 控制单轮时间 |
| 停滞 | 当前策略失效 | 切换策略重试 | 减少投入 |
| 下降 | 方向错了 | 回滚，换策略 | 最小投入 |

### 2.3 能量阶段

把时间看作能量，不同阶段行为不同：

| 能量 | 模式 | Agent 行为 |
|---|---|---|
| 100%-50% | 全力模式 | 所有 Agent 正常工作，大胆探索 |
| 50%-20% | 聚焦模式 | 只运行最低分相关 Agent，精准打击 |
| 20%-10% | 精简模式 | 只保留 4 个核心 Agent |
| <10% | 终结模式 | 停止优化，整理输出 |

### 2.4 人工介入时计时暂停

时间预算约束的是 Agent 工作时间，不是用户思考时间：

- 广度扫描后基线确认 → 暂停
- 关键决策点 → 暂停
- 最终输出审核 → 暂停

记录两种时间：Agent 时间（计入预算）和等待时间（不计入）。

### 2.5 预算自动估算与手动覆盖

```
/require "记账App"              → Agent 广度扫描后自动估算
/require "记账App" --time 120   → 手动指定 120 分钟
/require "记账App" --rounds 15  → 手动指定 15 轮
```

优先级：手动 > 自动。

## 三、多维评分体系

### 3.1 基础维度（默认启用）

| 维度 | 评什么 | 核心问题 |
|---|---|---|
| **完整性** | 功能点、异常场景、边界条件有没有漏 | "还缺什么？" |
| **一致性** | 需求之间有没有矛盾、冲突、重复 | "自己打脸了没？" |
| **可行性** | 需求是否合理、可落地、不脱离实际 | "真的能做吗？" |
| **用户旅程** | 用户旅程是否通顺、体验是否闭环 | "用户会不会卡住？" |
| **业务闭环** | 商业逻辑是否成立、核心流程是否走得通 | "这事儿能转起来吗？" |

每个维度 0-10 分，达标线默认 **7 分**（可配置）。

维度可扩展、可裁剪：
```
/require "后台系统" --dimensions +安全,-业务闭环
```

### 3.2 评分机制：主审 + 专业复核 + 人工校准

- **评估 Agent（主审）**：对所有维度独立打分
- **各专业 Agent（复核）**：只对自己擅长的维度打分
- **加权**：主审 40% + 专业复核 60%
- **人工校准**：仅在两个节点触发（广度扫描后 + 最终输出前）

复核对应关系：
| 维度 | 主审 | 复核 Agent |
|---|---|---|
| 完整性 | 评估 Agent | 完整性 Agent |
| 一致性 | 评估 Agent | 一致性 Agent |
| 用户旅程 | 评估 Agent | 用户旅程 Agent |
| 业务闭环 | 评估 Agent | 业务闭环 Agent |
| 可行性 | 评估 Agent | 可行性 Agent |

### 3.3 每次评分输出三样东西

```
评分 = 分数 + 置信度 + 理由

示例：
  完整性：6 分 | 置信度：高 | 理由：缺少支付失败的异常处理流程
```

置信度影响权重：高 ×1.0，中 ×0.7，低 ×0.4。

### 3.4 评分计算公式

最终分数 = 加权原始分 × 置信度系数 × 证据系数

```
示例：
  主审打分：7 分（置信度：高）
  复核打分：6 分（置信度：中）
  证据支撑：S 级

  加权原始分 = 7 × 0.4 + 6 × 0.6 = 6.4
  置信度系数 = (1.0 × 0.4 + 0.7 × 0.6) = 0.82
  证据系数 = 1.0

  最终分数 = 6.4 × 0.82 × 1.0 ≈ 5.2（偏保守，因为复核方置信度不高）
```

### 3.5 分歧检测

主审和专业 Agent 打分差距 ≥ 2 分时触发"辩论轮"：
- 两个 Agent 各自陈述理由
- 找出分歧根源（往往是需求描述模糊）
- 分歧本身作为"待优化项"注入下一轮

### 3.6 证据支撑评分

评估 Agent 打分时可查询知识库：
- 有 S 级证据支撑 → 证据系数 1.0
- 有 B 级证据支撑 → 证据系数 0.8
- 无证据 → 证据系数 0.6
- 与证据矛盾 → 证据系数 0.3 + 触发预警

### 3.7 跨维度一致性检查

每轮打分后额外检查维度之间的矛盾（如安全要求鉴权 vs 用户旅程要求无登录浏览）。

### 3.8 评分体系自身进化

人工校准时反向修正权重，评分标准随使用逐渐校准。

## 四、Agent 架构

### 4.1 总览

```
┌─────────────────────────────────────────────┐
│           Skill：总编排器（require.md）         │
└──────────────────┬──────────────────────────┘
                   │
    ┌──────────────┴──────────────────┐
    ▼                                 ▼
┌────────┐                      ┌──────────┐
│ 知识引擎 │                      │  共享黑板  │
└────────┘                      └─────┬────┘
                                      │
  基础维度          防护类        质量类       架构类
  完整性            安全          性能         数据
  一致性                         无障碍       依赖与集成
  用户旅程                       与国际化
  业务闭环
  可行性

  系统角色：红队 | 评估 | 写手

总计：14 个 Agent
```

### 4.2 各 Agent 详细定义

详见附录 A：[Agent 详细定义](./appendix-a-agent-details.md)

每个 Agent 包含：
- 核心职责
- 详细功能清单
- 策略池（含策略生成能力）
- 自我进化机制

### 4.3 Agent 分类与默认开关

**默认启用（所有项目）：**
- 完整性、一致性、用户旅程、业务闭环、可行性
- 安全、红队
- 评估、知识引擎、写手

**按需启用（系统自动建议）：**
- 性能 → 高并发、大数据量时
- 无障碍与国际化 → 面向全球或有合规要求时
- 数据 → 数据密集型项目
- 依赖与集成 → 涉及第三方服务时

支持自定义 Agent 插槽：用户可添加行业专属 Agent。

### 4.4 Agent 依赖链

优化顺序受依赖关系约束：

```
层级一（地基层）：完整性 → 数据
层级二（结构层）：一致性、用户旅程、业务闭环
层级三（防护层）：安全、依赖与集成
层级四（打磨层）：性能、无障碍与国际化、可行性
```

说明：可行性虽是基础维度之一，但放在打磨层是因为——可行性评估依赖其他所有维度的输出才能给出准确判断（功能不全时无法评估"能不能做"）。

深度优化选择 = 最低分优先 + 依赖条件满足。

### 4.5 Agent 协作机制

**黑板机制（分阶段读写）：**
- 阶段一（专注执行）：当轮被选中优化的 Agent 只写不读
- 阶段二（交叉审阅）：所有 Agent 只读，记待办
- 阶段三（红队+写手）：挑战 + 整合

**直接问答：** 阶段一中目标 Agent 可向其他 Agent 发起定向提问（每轮 ≤2 次，最大问答深度 2 层）。

**待办队列：** 每个 Agent 维护自己的待办队列，在自己的轮次中消化。

**话题归属去重：** 多 Agent 关注同一话题时，指定主责 Agent + 协作 Agent，避免重复。

**冲突升级机制：**
1. 自动协调（一方理由更充分 → 采纳）
2. 评估 Agent 仲裁
3. 升级人工决策

### 4.6 Agent 质量保障

- **预检**：依赖满足、信息具备、状态正常
- **后检**：质量门控（具体性、依据性、可操作性、去重）+ 范围检查 + 意图锚点检查
- **信任度评分**：动态追踪每个 Agent 的可靠性，影响建议权重
- **疲劳检测**：重复率 >50% 或连续空产出 → 刷新上下文
- **容错**：单 Agent 故障不中断流程，降级运行
- **版本管理**：Agent 进化有版本记录，表现下降可回滚

### 4.7 Agent 行为阶段自适应

| 阶段 | 发现门槛 | 建议粒度 | 策略倾向 |
|---|---|---|---|
| 广度扫描 | 低（宁多勿漏） | 粗（指方向） | 探索 |
| 深度前期 | 中（聚焦高影响） | 中（给方案） | 进攻 |
| 深度后期 | 高（只抓细节） | 细（精确措辞） | 防守 |
| 收敛期 | 不提新发现 | 查漏补缺 | 终结 |

### 4.8 红队 Agent 特殊设计

红队不只挑战需求，还挑战其他 Agent 的结论和评分结果。每 3 轮做一次全系统审计。攻击力可动态调节（温和/标准/激进）。

### 4.9 输出标准化

所有 Agent 写入黑板的内容遵循统一 JSON 格式：agent、round、type、target、severity、content、suggestion、evidence、related_agents、confidence。

## 五、知识引擎 Agent

独立成章，因其复杂度高且贯穿全流程。

### 5.1 定位：从搜索工具到情报分析中心

不是"你问我搜"的搜索工具，而是"自带嗅觉、持续积累、主动供给"的情报官。

### 5.2 三种工作模式

| 模式 | 触发时机 | 工作方式 |
|---|---|---|
| 前置侦察 | 优化开始前 | 正面+反面搜索，输出领域简报 |
| 主动嗅探 | 优化过程中 | 读黑板发现新话题，自动补充搜索 |
| 按需深挖 | 被 Agent 调用时 | 先查缓存，未命中才搜索 |

### 5.3 全球信息渠道

默认全球范围搜索，不限项目类型（App、后台、系统、平台等）。知识引擎根据项目内容自动判断去哪里搜、搜什么。渠道不写死，由引擎动态发现最佳信息源。

用户可缩窄：`--market cn`。

### 5.4 情报分析六步循环

```
① 采集（正面+反面+盲区补充）
② 清洗（来源分级 S/A/B/C + 去重）
③ 分析（找规律 + 检测冲突 + 识别盲区）
④ 生产（结论 + 冲突报告 + 盲区报告）
⑤ 投递（定向推送给最需要的 Agent）
⑥ 反馈（情报被用了吗？分数变了吗？→ 改进采集）
```

### 5.5 来源分级

| 等级 | 说明 | 处理方式 |
|---|---|---|
| S 级 | 官方文档、权威报告 | 直接写入知识库 |
| A 级 | 专业媒体分析 | 直接写入 |
| B 级 | 普通用户评论 | 标记 [待验证]，需交叉验证 |
| C 级 | 单一来源、无法验证 | 不写入，除非有佐证 |

### 5.6 关键能力

- **冲突即情报**：信息源矛盾 → 标记为决策点，推送给相关 Agent
- **盲区探测**：对照需求文档功能模块，主动发现知识空白
- **反馈闭环**：追踪情报是否被使用、是否提升了分数
- **信息保鲜**：每条信息带时间戳和保鲜期，跨项目复用时检查过期
- **隐私边界**：搜索前脱敏过滤，敏感信息不外传
- **交叉验证**：关键信息（法规、标准）至少 2 个独立来源确认
- **定向投递**：根据信息内容和维度标签推送给对应 Agent

### 5.7 边界约束

- 能量占比：最多消耗总时间预算的 20%
- 搜索去重：搜过的不重复搜
- 深度递减：前置侦察最深，后期能量低时停止嗅探
- 输出精简：写入知识库的是结构化摘要

### 5.8 降级方案

| 故障 | 处理 |
|---|---|
| 部分查询失败 | 用已有结果，失败的加入重试队列 |
| 大面积失败 | 切换无情报模式，通知 Agent |
| 搜到但质量差 | 等同于未搜到，不污染知识库 |

## 六、自进化机制

### 6.1 三层进化

```
第一层（轮内）→ 学会"这一轮什么管用"
  策略试错 → 保留有效 / 丢弃无效 / 生成新策略

第二层（轮间）→ 学会"这个项目什么管用"
  优化策略顺序、Agent 协作模式、评分权重

第三层（跨项目）→ 学会"什么类型的项目用什么策略"
  领域专精、通用洞察、反模式积累
```

### 6.2 策略生成（不只是选择）

当所有预置策略试完仍未达标时：
1. 分析已有策略的规律
2. 组合型 / 借鉴型 / 变异型 生成新策略
3. 试运行 → 有效则加入策略池

### 6.3 跨 Agent 知识迁移

提取通用洞察共享给所有 Agent（如"支付模块是高敏感区域"）。

### 6.4 领域专精档案

做过的领域自动积累专精档案（常见问题、最佳策略、领域术语），新项目匹配时自动加载。

### 6.5 反模式库

显式记录"什么不该做"——失败的策略、常见陷阱、领域特有的坑。

### 6.6 Agent 版本管理

Agent 进化有版本记录，连续 3 个项目表现不如旧版 → 自动回滚。

### 6.7 协作组合记忆

记录有效的 Agent 搭配（如安全+红队→支付模块），编排器自动复用。

### 6.8 进化数据健康

- 置信度：样本量不足的经验仅供参考
- 过期：超 6 个月未验证 → 降级为待验证
- 反向验证：经验导致表现变差 → 标记可能过时

### 6.9 编排器自身进化

调度逻辑、阶段划分、轮次分配模式也随项目积累而优化。

### 6.10 用户偏好学习

自动构建用户画像（关注维度、风格偏好），新项目自动应用。

## 七、模块化文档架构

### 7.1 文档结构

```
docs/requirements/{项目名}/
├── requirement-overview.md         ← 主文档（目录+概览，始终精简）
├── modules/                        ← 按功能模块拆分
│   ├── 01-user-system.md
│   ├── 02-accounting.md
│   └── ...
├── cross-cutting/                  ← 跨模块需求
│   ├── security.md
│   ├── performance.md
│   └── error-handling.md
├── traceability.md                 ← 追溯矩阵
├── executive-summary.md            ← 一页纸摘要
├── changelog.md
├── open-questions.md
└── report.md
```

### 7.2 自动拆分时机

- 功能模块 ≤3 个 → 单文件
- 功能模块 4-8 个 → 建议模块化
- 功能模块 >8 个 → 自动模块化
- 单文件超 50 页 → 触发拆分建议

### 7.3 模块依赖图

在 requirement-overview.md 中维护模块间依赖关系，修改某模块时自动提示下游影响。

### 7.4 模块生命周期

草稿 → 优化中 → 待审核 → 已审核 → 已锁定。已锁定的模块 Agent 不再触碰。用户可用 `/require:lock --module X` 和 `/require:unlock --module X` 手动控制。用户可用 `/require:split` 手动触发文档模块化拆分。

### 7.5 模块并行优化

无依赖的模块可同时由不同 Agent 优化。

### 7.6 模块粒度动态调整

模块过大（>30 页）→ 建议拆分；模块过小（<1 页）→ 建议合并。

### 7.7 两层评分

模块级评分（每模块×每维度）+ 全局评分（按模块权重加权汇总）。核心模块权重高。

### 7.8 需求优先级

每个功能点标注 P0-P3 优先级，基于业务闭环、用户旅程、可行性综合判断。

### 7.9 追溯矩阵

用户目标 → 功能模块 → 需求项 → 优先级，砍需求时可追溯影响。

### 7.10 多层级自动摘要

- 一句话摘要（给老板）
- 一页纸摘要 executive-summary.md（给管理层）
- 完整文档（给执行团队）

## 九、输入输出设计

### 9.1 三种输入方式

```
/require "描述"              ← 从想法开始
/require --file ./prd.md     ← 从文档开始
/require --resume            ← 从中断处继续
```

### 9.2 启动参数

| 参数 | 说明 | 默认值 |
|---|---|---|
| --time | 时间预算（分钟） | 自动估算 |
| --rounds | 轮次上限 | 自动估算 |
| --target | 达标分数线 | 7 |
| --market | 目标市场 | 全球 |
| --offline | 离线模式（知识引擎仅用 LLM 知识 + 进化日志，跳过前置侦察的联网搜索，搜索类策略替换为非搜索策略） | 否 |
| --budget | 成本预算：low（6 核心 Agent、搜索减半）/ medium（默认配置）/ high（全部 Agent、不限搜索） | medium |
| --template | 自定义模板路径 | 内置 |
| --output | 自定义输出目录 | ./docs/requirements/ |
| --dimensions | 调整评分维度 | 默认 5 个 |
| --agents | 调整 Agent | 默认启用组 |
| --collab | 团队协作模式 | 否 |
| --explain | 输出附带来源注释 | 否 |
| --project-template | 使用项目模板 | 无 |

### 9.3 输出格式自适应

| 输入阶段 | 输出格式 |
|---|---|
| 模糊想法 | 结构化大纲 |
| 初步需求 | 功能清单 |
| 成熟需求 | 完整 PRD |

Agent 自动判断阶段，格式随优化推进自动升级。

### 9.4 导出

```

/require:export json     ← 结构化 JSON
/require:export csv      ← 功能清单 CSV
```

## 十、多人协作

### 10.1 三种协作模式

**接力模式：** 一个人优化 → git push → 另一个人 `--resume --role developer` 继续。

**并行模式：** `--collab` 生成团队输入模板（team-input/），各角色填写后 `--merge-input` 汇总。共识直接纳入，分歧标记为待讨论。

**评审模式：** 团队成员在文档中用 `<!-- @review 姓名: 意见 -->` 标注，`--review` 读取标注继续优化。

### 10.2 并发保护

锁文件含心跳（30 秒更新）+ 自动过期 + 进程检测。如需强制接管已锁定的会话：`/require --force`。

### 10.3 模块化协作

模块化后可按模块分配负责人，不同人并行优化不同模块。

## 十一、用户体验

### 11.1 渐进式复杂度

| 使用次数 | 模式 | 可见功能 |
|---|---|---|
| 1-3 次 | 极简 | 6 个核心 Agent（完整性、一致性、用户旅程、业务闭环、红队、写手，评估和知识引擎后台运行），基本命令 |
| 3-10 次 | 标准 | 全部 Agent，进度详情 |
| >10 次 | 专家 | 所有高级功能 |

## 十二、系统健壮性

### 12.1 风险与对策总览

| 风险 | 对策 |
|---|---|
| 编排器崩溃 | 状态持久化 + 断点恢复 |
| 黑板损坏/过大 | 分片存储 + 原子写入 |
| 评分死循环 | 震荡检测 + 强制协作 |
| Agent 问答死锁 | 深度限制 + 死锁检测 |
| 进化学坏 | 置信度 + 过期 + 反向验证 |
| 上下文溢出 | 上下文预算管理（40% 文档 / 20% 黑板 / 15% 摘要 / 10% 策略与经验 / 10% 知识库 / 5% 预留） |
| 锁文件失效 | 心跳 + 自动过期 + PID 检测 |
| 回滚状态不一致 | 全状态同步回滚 |
| 质量门控卡死 | 打回限制 3 次 + 降级放行 |
| 配置损坏 | 校验 + 默认降级 + 自动备份 |
| 磁盘空间增长 | 自动清理策略 |
| 用户手动改文档 | 哈希检测 + 合并选项 |
| Agent 集体盲区 | 元审计 + 随机质疑（每 3 轮） |
| 红队过于激进 | 攻击力动态调节 |
| 假情报污染 | 交叉验证 + 溯源标记 |
| 用户反馈矛盾 | 冲突检测 + 决策日志 |
| 用户长时间不响应 | 磁盘持久化 + 会话无关恢复 |
| 需求范围剧变 | 变更影响评估 + 分支/重置选项 |
| Token 消耗爆炸 | 单次上限 + 异常检测 |
| 最终文档仍有问题 | 终审四步扫描 |
| Ctrl+C 强制退出 | 优雅关闭 + 脏状态恢复 |
| 无法排查问题 | 全链路日志 + trace 命令 |
| 插件依赖缺失 | 环境检查 + 降级运行 |
| 多项目并发 | 项目级工作区隔离 |
| 进化数据写入冲突 | 写入锁 + 追加模式 |
| 系统无法自查 | 自诊断命令 |
| Agent 文件被误改 | 完整性校验 |
| 需求膨胀 | 范围基线 + 膨胀率监控 |
| Agent 幻觉 | 意图锚点 + 偏离检测 |
| 异常需求 | 合理性检测 |
| 冷启动 | 内置种子经验 |
| 插件更新 | 版本迁移机制 |

### 12.2 环境检查

启动时自动检查：Claude Code 版本、必需工具、可选工具、磁盘空间、输出目录可写性。工具缺失时降级运行而非阻塞。

### 12.3 Git 集成（可选）

可配置在阶段完成、人工校准、最终输出时自动 git commit。

## 十三、完整流程

```
/require "记账App"
    │
    ▼
阶段零：系统初始化
    环境检查 → 加载配置/画像/经验 → 创建工作区
    → 输入预处理 → 提取意图锚点 → 预估预算 → 确定 Agent
    │
    ▼
阶段一：知识引擎前置侦察
    正反面搜索 → 来源分级 → 交叉验证 → 输出领域简报
    │
    ▼
阶段二：广度扫描
    并行：基础维度 Agent + 按需 Agent
    串行：红队 → 评估（基线评分）→ 写手（v1）
    → 建立范围基线 → 输入成熟度检测
    → ⏸️ 人工校准 #1
    │
    ▼
阶段三：深度优化循环
    每轮：选目标 → 选策略 → 预检 → 专注执行 → 交叉审阅
         → 红队挑战 + 写手整合 → 后检 → 评估 → 决策 → 状态更新
    循环直到终止条件触发
    │
    ▼
阶段四：终审
    通读一致性 → 歧义扫描 → 完整性核查 → 格式检查
    → ⏸️ 人工校准 #2
    │
    ▼
阶段五：交付输出
    生成最终文档 → 输出到 docs/ → 用户满意度反馈
    → 释放锁 → 提供下一步选项
    │
    ▼
阶段六：进化沉淀
    记录项目日志 → 更新策略/信任度/组合/领域/反模式
    → 清理工作区
```
